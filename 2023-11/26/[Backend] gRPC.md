### RPC
- Remote Procedure Call
- 원격 제어 위한 별도 코딩 없이 다른 공간에서 함수 & 프로시저 실행하게 하는 통신 기술

### 함수 vs 프로시저
#### 함수
- Input 에 따른 Output 발생을 목적 ( Return 값 필수 ! )
- Client 단에서 처리 ( 간단한 계산 및 수치 도출에 사용 )
#### 프로시저
- Output 목적 X
- 명령 단위가 수행하는 절차
- Return 값 존재 상관 없음 + Server단 처리 ( 큰 단위 실행 , 프로세싱 )

##### 어디에 어떻게 쓰는데?
- 일반적으론 , 프로세스는 자신 주소공간 내 , 함수만 호출
-> RPC 는 자신과 다른 주소공간 동작 프로세스의 함수를 실행하게 해줌
=> MSA 구조 서비스에서 , 언어 및 환경 구애받지 않고 비즈니스 로직 개발에 집중 가능

### RPC 목표
- Client - Server 간 소통에 필요한 상세정보는 감춘다. ( => 언어 , 환경 구애 받지 않음 )
- Client 와 Server 는 일반 메소드 호출하는 것처럼 원격지 프로시저 호출 가능

### RPC 동작 방식
#### IDL 기반
##### IDL 
- Interface Definition Language
- 한 언어에 국한되지 않은 언어 중립적 인터페이스
- 정의 언어로 , 구현 언어로 Mapping 지원

1. IDL 을 사용해 서버 호출 규약 정의
	- ( 함수명 , 인자 , 반환값 ) 에 대한 데이터 타입 저장된 IDL 파일을 rpcgen Compiler 이용해 Stub Code 자동 생성
2. Stub 은 원시소스 코드 ( C , Java ) 형태 만들어지므로 , Client - Server 에 포함해 Build
3. Client Program 입장에서 자신 프로세스 주소공간의 함수 호출하는 것으로 보이는데 , 
		클라이언트 stub 에 정의된 함수 호출
4. Stub 코드는 데이터 타입을 XDR 형식으로 변환해 RPC 호출 실행
##### XDR
- External Data Representation
- 기본 데이터 타입 ( 정수형 , 부동 소수점 ) 에 대한 메모리 저장 방식 ( Little/Big Endian ) 는 아키텍처 별로 다름
		+ 네트워크 전송 과정 중 바이트 전송 순서 보장하기 위함
5. 서버는 수신된 함수 / 프로시저 호출 대한 처리를 서버 Stub 통해 처리 완료 후 , 결과값을 XDR 변환해 return
6. 클라이언트 프로그램은 서버가 return 한 결과값을 전송 받음

### 장점
- 고유 프로세스 개발에 집중 가능 ( 하단 , 네트워크 프로토콜 신경쓰지 않아도 되기 때문 )
- 프로세스간 통신 기능 비교적 쉽게 구현하고 정교한 제어 가능

### 단점
- 호출 실행 과 반환 시간이 보장되지 않음! ( 네트워크 끊길 때 , 치명적 문제! )
- 보안이 보장되지 않음
